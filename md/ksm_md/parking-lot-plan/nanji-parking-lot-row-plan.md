# Nanji Parking Lot Row Plan

## 문서 목적
- `parking_lot` 테이블에 어떤 주차장 row를 넣을지 확정한다.
- 현재 ML 예측 단위와 실제 서비스에서 보여줄 대체주차장 단위를 분리해서 정리한다.
- 이후 DB 입력, FastAPI import, 앱/웹 UI 연결의 기준 문서로 사용한다.

## 왜 이 문서가 필요한가
- 현재 ML 데이터는 `개별 주차장`이 아니라 `난지 묶음 단위` 기준이다.
- 반면 앱과 관리자 웹에서는 사용자에게 실제로 보여줄 대체주차장이 필요하다.
- 따라서 `예측 기준 주차장`과 `사용자에게 보여줄 주차장`을 분리해서 설계해야 한다.

## 현재 확인된 데이터 기준
- `hmw` 원본 데이터에서 난지 관련 주차장명은 아래 2개만 직접 확인된다.
  - `난지1,2,3,4주차장`
  - `난지1,2,3주차장`
- 즉, 현재 예측 모델은 개별 주차장 예측이 아니라 묶음 단위 예측으로 보는 것이 안전하다.

## 추천 방향
- `parking_lot`에는 실제 서비스에 보여줄 주차장 4개를 넣는다.
- 예측 데이터는 당분간 `난지 메인 주차장` row에만 연결한다.
- 대체주차장 3개는 정보 제공과 우회 안내 중심으로 운영한다.

## 최종 추천 row 4개

### 1. 난지 메인 주차장
- 역할:
  - 난지 묶음 예측 데이터를 연결할 대표 row
  - 현재 예측, 추천 방문 시간, 출발 타이밍 추천의 중심 주차장
- 사용 이유:
  - ML 결과가 난지 묶음 단위이므로, 예측 연결용 대표 엔터티가 필요하다.

### 2. 난지 캠핑장 주차장
- 역할:
  - 대체주차장 안내용
  - 지도, 즐겨찾기, 알림 등의 사용자 기능 대상

### 3. 하늘공원 주차장
- 역할:
  - 대체주차장 안내용
  - 혼잡 시 우회 선택지 제공

### 4. 월드컵공원 주차장
- 역할:
  - 대체주차장 안내용
  - 지도/즐겨찾기/대체안내 중심

## 권장 입력값 초안

| 구분 | p_name | p_display_name | 역할 | 예측 연결 | 대체주차장 노출 |
|---|---|---|---|---|---|
| 1 | 난지메인 | 난지 메인 주차장 | 난지 묶음 예측 대표 row | 예 | 예 |
| 2 | 난지캠핑장 | 난지 캠핑장 주차장 | 대체주차장 | 아니오 | 예 |
| 3 | 하늘공원 | 하늘공원 주차장 | 대체주차장 | 아니오 | 예 |
| 4 | 월드컵공원 | 월드컵공원 주차장 | 대체주차장 | 아니오 | 예 |

## 기능 플래그 추천

### 난지 메인 주차장
- `p_supports_realtime_congestion = 1`
- `p_supports_prediction = 1`
- `p_supports_departure_timing = 1`
- `p_supports_map_view = 1`
- `p_supports_favorite = 1`
- `p_supports_notification = 1`

설명:
- 현재 메인 주차장은 서비스 중심 엔터티로 사용한다.
- 예측과 추천은 이 row에만 먼저 연결한다.

### 대체주차장 3개
- `p_supports_realtime_congestion = 0` 또는 실제 데이터 확보 시 `1`
- `p_supports_prediction = 0`
- `p_supports_departure_timing = 0`
- `p_supports_map_view = 1`
- `p_supports_favorite = 1`
- `p_supports_notification = 1`

설명:
- 현재 데이터 기준으로 대체주차장 3개는 예측 대상이 아니다.
- 대신 지도, 즐겨찾기, 대체안내 같은 사용자 기능은 유지한다.

## 설계 이유

### 왜 `난지전체` 하나만 두지 않는가
- 앱에서는 실제로 대체주차장을 사용자에게 보여줄 필요가 있다.
- 즐겨찾기, 지도 보기, 대체안내는 개별 주차장 단위가 더 자연스럽다.
- 따라서 대표 row 하나만 두면 UX가 너무 뭉뚱그려진다.

### 왜 대체주차장 3개에 예측을 바로 붙이지 않는가
- 현재 ML 데이터는 개별 주차장으로 분리되지 않았다.
- 묶음 예측을 근거 없이 3개 주차장에 나눠 넣으면 데이터 왜곡이 생긴다.
- 지금 단계에서는 메인 주차장에만 예측을 연결하는 것이 안전하다.

## FastAPI 연결 기준

### 예측 import
- 현재 가공된 ML 파일은 `난지전체` 기준 예측이다.
- FastAPI import 시에는 이 값을 `난지 메인 주차장` row의 `p_id`에 연결한다.
- 즉 `parking_prediction.pp_parking_lot_id`는 `난지 메인 주차장`을 가리키게 한다.

### 대체주차장 응답
- 앱/웹에서는 `난지 메인 주차장` 상세 또는 추천 화면 아래에 대체주차장 3개를 같이 내려준다.
- 대체주차장 3개는 현재 예측이 아니라 위치/안내 중심 데이터로 사용한다.

## 이후 확장 방향
- 나중에 개별 주차장 단위 데이터가 생기면 각 대체주차장에도 예측을 붙인다.
- 그때는 아래처럼 확장 가능하다.
  - `난지 캠핑장 주차장` 예측 연결
  - `하늘공원 주차장` 예측 연결
  - `월드컵공원 주차장` 예측 연결

즉 현재 구조는 임시 방편이 아니라, 나중에 세분화된 예측 데이터가 들어와도 확장 가능한 구조다.

## 최종 결론
- `parking_lot`에는 4개 row를 넣는다.
  - `난지 메인 주차장`
  - `난지 캠핑장 주차장`
  - `하늘공원 주차장`
  - `월드컵공원 주차장`
- 예측은 당장은 `난지 메인 주차장`에만 연결한다.
- 대체주차장 3개는 사용자용 안내 구조로 유지한다.
- 이 방식이 현재 ML 데이터 한계와 서비스 UX를 가장 안정적으로 같이 만족시킨다.
